## 4. Findings

**F-001 — Critical — `RFQ-0102` document_id collision**
A different, already-committed document also declares `document_id: RFQ-0102` — "Sample Dependency Test - Structural Integration" (Grok, v0.1, `drafts/RFQ-0102-Sample-Dependency-Test/`). Two unrelated documents cannot share one identifier under RFQ-0100 P2 (Repository Authority) and P5 (Traceability). Requires a Human Authority decision — e.g. renumber the sample test doc, or confirm it should be retired — before v0.1.1 can be committed as the authoritative `drafts/RFQ-0102/RFQ-0102.md`.

**F-002 — Critical — Frontmatter fails A1 schema validation**
Programmatically validated against `schemas/meridian-a1-markdown-artefact-standard.yaml` (Draft 2020-12, `additionalProperties: false`) using a JSON Schema validator. 9 violations: missing required fields `date`, `programme`, `author`; disallowed fields present instead — `schema_version`, `date_created` (should be `date`), `authors` (should be singular `author`), `review_layers`, `approval`, `parent_documents`. Per RFQ-0100 §10, this is exactly the class of error the CI A1 validation gate is meant to hard-fail on.

**F-003 — Significant — Review artefact doesn't conform to the mandatory A1 Review Artefact Standard**
RIP-003A §8 requires review artefacts to use specific YAML frontmatter, a Review Scope, a Document Under Review table, a RIP-1 Alignment Assessment, severity-tagged findings (`F-### — Title`), a Summary table, and a Disposition. The supplied "Gemini Layer 2B Technical Review" uses none of this — informal header, unstructured "Pass" checklist, no finding IDs, no severities, no disposition. GEMINI.md explicitly instructs Gemini to "maintain exact frontmatter... A1 schema... verify outputs against schemas," so this is a deviation from Gemini's own onboarding instructions, not an ambiguous spec.

**F-004 — Critical — `models.py`: `Placeholder` class crashes at import (verified by execution)**
`TypeError: non-default argument 'name' follows default argument`. Giving `type` a default value in the `Placeholder(Node)` subclass, combined with `slots=True` inheritance, breaks dataclass field ordering. The module fails to import, so nothing downstream that imports it — including `graph.py` — can run. Verified fix: mark the subclass `kw_only=True`.

**F-005 — Critical — `graph.py`: `export_dict()` crashes on every call (verified by execution)**
`TypeError: vars() argument must have __dict__ attribute`. `Node`, `Edge`, and `Placeholder` all use `@dataclass(slots=True)`, so instances have no `__dict__`; `vars()` cannot work on them. This directly blocks RFQ-0102 §13's acceptance criterion "graph export formats are supported." Verified fix: use `dataclasses.asdict()` instead. Forward note for the not-yet-built `exporter.py`: `datetime` fields aren't JSON-serializable by default (`json.dumps(..., default=str)` will be needed); the `str`-mixin enums already serialize correctly with no special handling.

**F-006 — Significant — No migration plan for overlapping legacy scripts**
`scripts/dependency_resolver.py` (already committed, functioning) already parses A1 `dependencies:` blocks and builds a suggestion graph — the same job RFQ-0102's new `resolver.py`/`parser.py`/`graph.py` package is designed to do. `predictive_gap_filler.py`, `ecosystem_synergy_scanner.py`, and `tier0_quality_gate.py` also overlap by name with the proposed `predictor.py`, `synergy.py`, and `scorer.py`. Neither RFQ-0102 nor Gemini's review states whether the new package supersedes, absorbs, or coexists with these.

**F-007 — Operational — Branch naming non-compliant, and the governing standards disagree with each other**
`draft/RFQ-0102-dependency-test` doesn't match RFQ-0100 §8's required `draft/<document_id>` pattern. Separately, RFQ-0100 §8 defines review branches as `review/<id>`, while RIP-003A §6 defines them as `review/<Document-ID>-ReviewerA` — the two standards conflict, and the two existing `review/*` branches in the repo follow the older RIP-003A pattern even though RFQ-0100 (status: Approved) is the later, formally approved document.

**F-008 — Minor — Edge dedup keys use enum repr, not value**
`add_edge()`'s dedup key embeds `f"{edge.type}"`, which renders as `"EdgeType.REQUIRES"` rather than `"requires"` (verified in this Python version). Deduplication still works correctly since it's applied consistently — readability nit only. Suggested: `edge.type.value`.
