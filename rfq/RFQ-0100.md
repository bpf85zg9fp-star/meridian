# Meridian GitHub Repository Bootstrap and Operationalisation Standard

## RFQ-0100 v0.2

-----

## 1. Purpose

This standard establishes the requirements for bootstrapping, structuring, governing, and operationalising the Meridian GitHub repository.

The repository shall serve as the single authoritative source of truth for all Meridian governance artefacts, standards, implementation specifications, and operational records.

This document marks the transition of Meridian from ad-hoc governance drafting into a controlled, versioned repository environment.

-----

## 2. Objectives

The repository implementation shall:

- Provide a single authoritative source for all Meridian artefacts;
- Enforce structural and formatting consistency;
- Support effective collaboration between human and AI participants;
- Prevent uncontrolled repository drift;
- Enable scalable and auditable governance processes;
- Establish automated validation and quality gates; and
- Fully operationalise the Meridian Review Grid.

-----

## 3. Definitions

- **Meridian** — The overarching governance programme.
- **Continuum** — The automation and orchestration layer governed by this repository.
- **Review Grid** — The multi-layer AI + human review pipeline defined in RIP-003A.

## 5. Confirmed Gaps Addressed

This standard resolves the following previously identified deficiencies:

- Absence of automated YAML/frontmatter validation;
- Lack of document identifier uniqueness enforcement;
- Undefined branch protection rules;
- Missing Review Grid operational triggers;
- Unclear delineation between GitHub Actions and n8n responsibilities;
- Absence of explicit six-domain repository structure; and
- Lack of formal Continuum governance provisions.

-----

## 6. Repository Scope

The Meridian repository governs programme-level artefacts, domain-specific standards (six domains), repository implementation specifications, Continuum automation, review pipelines, templates, schemas, and operational assets.

-----

## 7. Repository Structure

### 7.1 Top-Level Structure

```bash
/
├── rfq/
│   ├── estate/
│   ├── farm/
│   ├── village/
│   ├── resort/
│   ├── marina/
│   └── aviation/
├── rip/
├── repo/
├── reviews/
├── continuum/
│   ├── requirements/
│   ├── pipeline/
│   ├── domains/
│   ├── integration/
│   └── architectures/
├── schemas/
├── templates/
├── archive/
└── .github/
```

### 7.2 RFQ Domain Structure

Programme-wide artefacts reside at `/rfq/`. Domain-specific artefacts go into their respective subdirectories. Ad-hoc domain creation is prohibited without governance approval.

| Domain                  | Directory             | Scope |
|-------------------------|-----------------------|-------|
| Programme-wide          | `/rfq/`               | Cross-cutting RFQ artefacts |
| Estate                  | `/rfq/estate/`        | Estate domain |
| Farm                    | `/rfq/farm/`          | Farm domain |
| Village                 | `/rfq/village/`       | Village domain |
| Resort                  | `/rfq/resort/`        | Resort domain |
| Marina                  | `/rfq/marina/`        | Marina domain |
| Aviation Infrastructure | `/rfq/aviation/`      | Aviation domain |

### 7.3 Directory Purposes

| Directory      | Purpose |
|----------------|---------|
| `rfq/`         | Requirements and Framework Standards (organised by domain) |
| `rip/`         | Requirements Interchange Protocols and pipeline standards |
| `repo/`        | Repository implementation and status artefacts |
| `reviews/`     | All Layer 2A/2B/3 review artefacts |
| `continuum/`   | Continuum automation governance |
| `schemas/`     | YAML schemas and frontmatter definitions |
| `templates/`   | Approved document templates |
| `archive/`     | Deprecated/superseded artefacts |
| `.github/`     | Workflows, PR/issue templates |
