# RFQ-0100 — Repository Structure and Governance Architecture Standard

## 1. Purpose

This document defines the authoritative repository architecture, directory structure, document lifecycle rules, branch model, and governance workflows for the Meridian programme. It is the single source of truth for how all Meridian governance artefacts are organised, promoted, and maintained in the `bpf85zg9fp-star/meridian` GitHub repository.

All Meridian governance artefacts — including RFQ, RIP, financial governance (FIN), review artefacts, schemas, automation specifications, and AI pipeline onboarding documents — must conform to this standard.

## 2. Objectives

- Establish a single source of truth for all Meridian programme artefacts.
- Ensure auditability of all governance decisions and document state transitions.
- Enforce the multi-model AI review pipeline defined in RIP-003A before any artefact reaches Approved status.
- Define a controlled promotion lifecycle requiring Human Authority approval at the Approved gate.
- Provide a stable, traceable repository structure that supports all current and future Meridian governance activities.

## 3. Governance Principles

### P1 — Human Authority Supremacy

All governance decisions affecting Meridian artefacts require explicit Human Authority approval from the Estate Principal (Kevin) before taking effect. No AI pipeline layer output, automated CI result, peer review artefact, or Layer 3 Consolidation Package may substitute for this approval. The Estate Principal holds sole final approval authority over all governance artefacts under this programme.

### P2 — Repository Authority

The `bpf85zg9fp-star/meridian` repository is the single authoritative source of truth for all Meridian programme governance artefacts. No other repository, document store, local file, or AI session output supersedes the committed repository state. Artefacts not committed to this repository are not authoritative, regardless of their origin or format.

### P3 — Reviewer Independence

Each AI review pipeline layer (Layer 2A: Claude; Layer 2B: Gemini) must conduct its review in isolation, without reference to the outputs of other pipeline layers. Reviews must be produced as formal artefacts per the A1 Artefact Standard and committed to `reviews/<document_id>/`. No cross-contamination between pipeline layers is permitted at any stage.

### P4 — Auditability

All governance decisions, document state transitions, and Human Authority approvals must be recorded and retrievable. Audit records are maintained across commit history, pull request records, the `reviews/` directory, and `CHANGELOG.md`. The repository structure must support this requirement at all times.

### P5 — Traceability

Every governance artefact must carry a document identifier, version, status, author, date, and lineage reference in its A1-compliant YAML frontmatter. The `related_documents` field must identify all documents materially affected by or dependent on the artefact. Changes between versions must be documented in the `version_history` field.

### P6 — Controlled Promotion

No artefact may advance from Draft to Approved status without completing all required pipeline review stages and receiving explicit Human Authority approval. No bypass of any review stage is permitted under any circumstances, including urgency, operational pressure, or Estate Principal request outside the formal approval gate.

### 4.1 Directory Naming Convention

Directories at root level use lowercase naming (e.g., `rfq/`, `reviews/`, `continuum/`). Uppercase root directories (`ARCHIVE/`, `AUTOMATION/`, `MASTER_CHECKLISTS/`, `POLICIES/`) are legacy names retained for backward compatibility pending remediation. All new directories created under this standard must use lowercase.

## 5. Document Lifecycle

All Meridian governance artefacts follow a five-stage lifecycle. No artefact may skip a stage.

| Stage | Entry Criteria | Exit Criteria | Responsible Party | Artefacts Produced |
|-------|---------------|---------------|-------------------|--------------------|
| **Draft** | Artefact created under `drafts/<document_id>/` on branch `draft/<document_id>` | Layer 1 author declares the artefact ready for pipeline review | Layer 1 (ChatGPT) | A1-compliant draft document |
| **Review** | PR raised from `draft/<document_id>` to `review/<document_id>` branch | All pipeline layers (2A, 2B, 3) have produced review artefacts; Layer 3 Consolidation Package is complete | Layers 2A, 2B, 3 | Review artefacts in `reviews/<document_id>/`; Layer 3 Consolidation Package |
| **Approved** | Layer 3 Consolidation Package delivered to Kevin | Kevin grants explicit approval and records it in the PR and CHANGELOG.md | Kevin (Estate Principal) | Approval record; CHANGELOG.md entry |
| **Published** | Kevin approves merge of review branch to `main` | Artefact committed to `main` in its authoritative directory; status field updated to "Published" | Repository admin | Committed published document; updated CHANGELOG.md |
| **Archived** | Superseded by a later approved version | Prior version moved to `archive/<document_id>/`; status field updated to "Archived" | Repository admin | Archived copy; CHANGELOG.md entry |

## 6. Draft Rule

All new artefacts MUST be created under `drafts/<document_id>/` where `<document_id>` is the artefact's assigned identifier (e.g., `drafts/RFQ-0101/`).

The draft file MUST be named `<document_id>.md` (e.g., `drafts/RFQ-0101/RFQ-0101.md`) and MUST conform to the A1 Artefact Standard (`schemas/meridian-a1-markdown-artefact-standard.yaml`).

A draft MUST NOT be committed directly to `main`. All draft work MUST occur on a branch named `draft/<document_id>` (e.g., `draft/RFQ-0101`).

## 7. Review Rule

All review artefacts MUST follow the isolation model defined in RIP-003A. Each pipeline reviewer must conduct their review without reference to any other layer's review output.

Review artefacts MUST be placed in `reviews/<document_id>/` using the naming convention `<document_id>-<version>-<Reviewer>.md` (e.g., `reviews/RFQ-0101/RFQ-0101-v0.3-Claude.md`).

Review work MUST occur on a branch named `review/<document_id>` (e.g., `review/RFQ-0101`).

## 8. Branch Model

| Branch Pattern | Protected | Purpose | Creation Authority | Merge Requirement | Delete After Merge |
|----------------|-----------|---------|-------------------|-------------------|--------------------|
| `main` | Yes | Authoritative published artefacts | Repository admin only | PR approved by Kevin; all CI status checks passing | No |
| `archive` | Yes | Superseded archived content | Repository admin only | PR approved by Kevin | No |
| `draft/<id>` | No | Active drafting for a specific artefact | Any pipeline participant | PR to `review/<id>` upon Layer 1 completion | Yes |
| `review/<id>` | No | Consolidated review package for a specific artefact | Layer 3 (Grok) | PR to `main` upon Kevin approval | Yes |

Branch naming MUST use the `<id>` segment equal to the artefact's `document_id` value (e.g., `draft/RFQ-0101`, `review/RFQ-0101`). No freeform or arbitrary branch names are permitted under the `draft/` or `review/` prefixes.

## 9. Promotion Rules

Promotion between lifecycle stages requires the following actions. No stage may be bypassed and no exceptions are recognised.

**Draft → Review:** The Layer 1 author creates a PR from `draft/<document_id>` targeting `review/<document_id>`. The PR description must state the artefact is ready for pipeline review and identify the document_id, version, and submitting layer. No approval is required at this transition.

**Review → Approved:** Layer 3 (Grok) produces a Consolidation Package synthesising all pipeline findings and presents it to Kevin as the Human Authority gate. Kevin records approval by merging the PR or by leaving an explicit approval comment in the PR record. Approval MUST also be recorded in `CHANGELOG.md` with the date, document_id, version, and approver name.

**Approved → Published:** The repository admin merges the review branch to `main`. The artefact is moved from `drafts/<document_id>/` to its authoritative directory (e.g., `rfq/` for RFQ documents). The `status` field is updated to "Published". `CHANGELOG.md` is updated. The review branch is deleted.

**Published → Archived:** When superseded by a later approved version, the prior published version is copied to `archive/<document_id>/`. The `status` field in the archived copy is updated to "Archived". `CHANGELOG.md` is updated. The authoritative directory retains only the current published version.

## 10. Enforcement

CI/CD automation enforces this standard on all pull requests targeting `main` or `review/*` branches. All CI checks must pass before a PR may be merged.

**A1 Schema Validation:** Validates that the artefact's YAML frontmatter conforms to `schemas/meridian-a1-markdown-artefact-standard.yaml`. All required fields must be present and correctly typed. A frontmatter parse failure or required-field absence fails this check.

**Governance Policy Check:** Validates that the artefact resides in the correct directory for its lifecycle stage and document taxonomy; branch naming follows Section 8 conventions; and the `review_stage` field matches the current pipeline state.

CI workflow files are maintained in `.github/workflows/`. Functional implementation specifications are maintained in `AUTOMATION/CI-SPEC-001.md` (see Section 11). Current CI workflows are stubs pending functional implementation; this standard is considered partially operational until implementation is complete.
